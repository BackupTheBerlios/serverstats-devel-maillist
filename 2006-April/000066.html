> Schreibe es derzeit sozusagen auf eine Client/Server Variante um. Ziel ist es, auf einem Server einen Xinetd mit "rrdtool -" (in einer chroot) laufen zu haben, und die Commands der anderen via netcat an diesen zu schicken.

Client/Server-Support hatte ich mir auch schon überlegt, allerdings mit
einem etwas anderen Ansatz:
Der Server hat Serverstats laufen. Anstatt der update.php gibt es ein
zusätzliches Script fetchValues.php (o.ä.). Dieses Script wird vom
entfernten Rechner direkt per HTTP aufgerufen. Es gibt alle oder nur
ausgewählte Sourcen in XML kodiert zurück, die der Server dann
auswertet. Alternativ wird einfach ein serialisiertes Sources-Objekt
zurückgegeben bei dem die Daten schon abgerufen wurden (viel einfacher).

Was jetzt schon sehr einfach möglich ist wäre der Zugriff auf die Daten
(/proc etc.) oder auf die RRD-Dateien über einen Netzwerkmount. Hier
könnte NFS oder auch SSHFS zum Einsatz kommen. Ich denke wenn man das
mit SSHFS auf die RRD-Dateien anwendet hat man ein einfaches und
sicheres (verschlüsselt) System.

> Da sich meine virtuellen xen-interfaces (z.B. "vif10.0", "vif3.2", usw.) und die dazugehörigen Bridges ("xen-br0", "xen-br1", usw.) nicht mit traffic_proc auswerten ließen musste ich die Regex ein ganz klein wenig anpassen...
> Habe nun auch im ersten Teil (vor dem Doppelpunkt) Punkte und Bindestriche gestattet...
> if (preg_match('/([\w\d\.\-]+):\s*(\d+)\s+(\d+)\s+\d+\s+\d+\s+\d+\s+\d+\s+\d+\s+\d+\s+(\d+)\s+(\d+)\s+\d+\s+\d+\s+\d+\s+\d+\s+\d+\s+\d+/', $line, $m))
> Nun klappts auch mit solchen Interfaces ;)
> Da ich keine Nachteile an dieser angepassten Regex entdecken konnte, möchte ich vorschlagen diese Änderung in die nächste Release aufzunehmen.

Ich hab das schonmal ins SVN übernommen, heißt es ist auf dem besten Weg
ins nächste Release.
Vielen Dank für den Hinweis.
